# 랭버디(LangBuddy) 심층 진단 보고서

> 작성일: 2026-07-04
> 대상: React 19 + Vite + Supabase + Vercel serverless + Google Gemini 기반 영어 학습 웹앱
> 방법론: 9개 서브시스템 병렬 정독 + 학습과학 분석 + 전체 UX 여정 리뷰 → 객관적 주장 100건을 독립 검증관이 "반박 시도"로 적대 검증(CONFIRMED 100 / REFUTED 8) → 완결성 비평. 아래 결함은 모두 실제 코드 경로로 확인된 것입니다.

---

## 0. 한 줄 결론

**핵심 학습 루프(콘텐츠 → 단어 탭 → 저장 → 복습)의 "앞단"은 잘 설계되어 있으나, "뒷단(복습·기록·재방문)"이 구조적으로 무너져 있어 사용자의 실력이 늘지 않는다.** 구체적으로 ① 복습이 진짜 간격 반복(SRS)이 아니고, ② 학습 진척이 어디에도 기록되지 않아 통계가 영구히 0으로 표시되며, ③ 앱 전체에 "영어를 산출(output)하는" 지점이 없고, ④ 재방문을 유도하는 장치(스트릭·리마인더·이어보기)가 전무하다. 여기에 ⑤ 서버리스 API 전체가 무인증으로 열려 있어 API 키 유출·비용 폭탄 위험이 상존한다.

## 1. 사용자 학습이 안 되는 치명적 이유 (근본 원인)

학습과학(간격 반복, 인출 연습, comprehensible input, 동기 설계) 관점에서 가장 치명적인 것부터.

### 1.1 간격 반복이 사실상 존재하지 않는다 — README의 "SM-2"는 허위
- README는 "SM-2 알고리즘 복습 시스템"을 표방하지만, 실제 구현은 자체 "stack" 방식이다. 간격이 **1/2/3/7일 4단계로 고정**되고 **최대 간격이 영원히 7일**이다.
- 간격 반복의 핵심 효용은 성공할수록 간격이 지수적으로 확장(1→3→7→16→35→90일…)되어 장기기억을 굳히면서 일일 복습 부담을 일정하게 유지하는 것인데, 7일 상한이면 단어가 늘수록 **일일 복습량이 선형으로 폭증**해 사용자가 복습을 포기한다.
- 더 치명적으로 `stack`이 `-10`에 도달하면(약 10회 정답) 항목이 쿼리에서 **영구 제외**된다. 망각 곡선상 복습 없는 기억은 소멸하므로, 이것은 "10번 맞히면 그 단어를 잊게 만드는" 설계다.
- 스키마에 SM-2용 `ease_factor(2.5)`/`repetitions` 필드가 생성 시 기록되지만 이후 어디서도 갱신·사용되지 않는 **죽은 필드**다.
- 근거: [src/services/review.js:85-95](src/services/review.js#L85-L95)(간격 상한 7일), [src/services/review.js:48](src/services/review.js#L48)(영구 제외), [src/services/annotation.ts:79-80](src/services/annotation.ts#L79-L80)(미사용 필드 생성).

### 1.2 인출 연습(retrieval practice)이 강제되지 않고, 산출(production) 방향 카드가 전무
- Review 화면에서 평가 버튼(I know / I don't know)이 **카드가 뒤집히기 전부터 항상 렌더링**되어, 답을 확인하지 않고 'I know'를 연타해 세션을 끝낼 수 있다. 복습이 인출 연습이 아니라 클릭 노동이 된다.
- 카드 방향이 "영어 → 한국어 뜻" 인식(recognition) **단방향뿐**이다. 뜻→영어 산출, 타이핑, cloze(빈칸) 같은 능동적 산출 형식이 전혀 없다. 인식은 산출보다 처리 수준이 얕아 실제 사용 능력으로 전이되지 않는다.
- 단어 저장 시 **그 단어가 등장한 원문 문장(context)이 저장되지 않아**(word/definition/phonetic만 저장) 카드 앞면이 맥락 없는 고립 단어다.
- `ai_analysis_json`이 없는 highlight도 무조건 review_item이 생성되는데, 이 경우 Flashcard 뒷면이 앞면과 **동일한 텍스트**를 보여줘 "질문=답"인 무의미한 카드가 만들어진다.
- 근거: [src/pages/review/Review.jsx:141-154](src/pages/review/Review.jsx#L141-L154), [src/containers/flashcard/Flashcard.jsx:26,57-63](src/containers/flashcard/Flashcard.jsx#L57-L63), [src/components/modals/useWordLookup.js:60-65](src/components/modals/useWordLookup.js#L60-L65), [src/services/annotation.ts:40-42](src/services/annotation.ts#L40-L42).

### 1.3 동기·피드백 루프가 완전히 죽어 있다 — 통계가 항상 0, 스트릭/목표/리마인더 부재
- `updateTodayStats` 함수는 정의만 있고 **앱 어디서도 호출되지 않아** `user_stats` 테이블에 데이터가 쌓이지 않는다 → 주간 차트가 영원히 빈 막대.
- 'Reviewed' 통계는 `.gt('repetitions', 0)` 조건인데 `repetitions`는 0으로 박제되고 갱신되지 않으므로 몇 주를 복습해도 **'Reviewed 0, Rate 0%'**가 표시된다. 노력이 수치에 반영되지 않는 것은 무동기(amotivation)를 직접 유발한다.
- 스트릭, 일일 목표, 푸시/이메일 리마인더가 코드에 없다(`streak` grep 0건). 심지어 `notification_settings`·`push_tokens` 테이블은 스키마만 만들고 중단된 **미완성 데드 스키마**다.
- 근거: [src/services/stats.js:82](src/services/stats.js#L82)(미호출), [src/services/stats.js:24](src/services/stats.js#L24), [src/containers/stats-dashboard/StatsDashboard.jsx:60-72](src/containers/stats-dashboard/StatsDashboard.jsx#L60-L72), [supabase-schema.sql:156](supabase-schema.sql#L156).

### 1.4 AI "튜터"의 기본값이 한국어 설명 봇 — 영어 산출 기회가 설계상 배제됨
- AI 채팅 기본 시스템 프롬프트가 "반드시 한국어로만 답변해주세요"이고 기본 설정도 `AI_CHAT_LANGUAGE='Korean'`이다. 앱의 유일한 상호작용 채널이 "영어에 대해 한국어로 설명받는" 메타 대화이지, 영어 사용 연습이 아니다.
- 사용자가 영어로 쓰도록 유도하는 과제나 오류 교정 피드백 루프가 없다(교정 지시는 음성 conversationMode에만 존재). 채팅은 소스 주제로 제한되어 자유 회화 연습조차 막혀 있다.
- 저장되는 "뜻"조차 **비공식 Google Translate 엔드포인트(gtx)**로 기계번역한 문자열이라 품질 보장이 없다.
- 근거: [src/services/ai/chat.js:188-189](src/services/ai/chat.js#L188-L189), [src/services/settings.js:13](src/services/settings.js#L13), [src/hooks/useChat.js:61](src/hooks/useChat.js#L61), [src/services/ai/vocabulary.js:8-9](src/services/ai/vocabulary.js#L8-L9).

### 1.5 복습 루프의 신뢰를 깨는 데이터 무결성 버그
- **카운트 불일치**: 홈 ReviewCard는 `stack ≤ -10` 제외 조건이 없어 "N개 복습 있음"을 보여주지만, 실제 세션은 그 항목을 제외한다 → "복습 5개 있음"을 눌렀는데 즉시 빈 완료 화면. 앱이 고장난 것처럼 보인다.
- **UTC 타임존**: 날짜를 `new Date().toISOString().split('T')[0]`(UTC)로 계산해, KST 사용자는 자정~오전 9시에 오늘 도래한 카드가 안 보이고, interval 1일 카드가 같은 날 다시 등장한다. 아침 학습 습관을 만들려는 사용자에게 치명적.
- **'Review Again' 허위 버튼**: 복습을 다 끝내면 뜨지만 눌러도 due 항목만 조회하므로 빈 화면 → 초과 학습을 원하는 동기 높은 사용자를 좌절시킨다.
- 근거: [src/services/review.js:57-68](src/services/review.js#L57-L68) vs [:48](src/services/review.js#L48), [src/services/review.js:35,105](src/services/review.js#L35), [src/components/cards/ReviewCard.jsx:46-52](src/components/cards/ReviewCard.jsx#L46-L52).

### 1.6 난이도 조절(i+1)과 학습자 프로파일링이 없다
- 학습자 수준(i)을 측정하지도, 콘텐츠 난이도를 평가하지도 않는다. 온보딩은 레벨 테스트/목표 설정 없이 기능 설명 5장을 넘기는 것이 전부다.
- `ENGLISH_LEVEL` 설정은 AI 프롬프트 문구에만 쓰일 뿐 소스 추가·추천·정렬 어디에도 반영되지 않는다. 너무 어려운 자료면 인지 과부하로 이탈, 너무 쉬우면 학습이 안 일어나는데 앱은 어느 쪽도 감지·안내하지 못한다.
- 근거: [src/pages/onboarding/Onboarding.jsx:5-69](src/pages/onboarding/Onboarding.jsx#L5-L69), [src/services/settings.js:8,15](src/services/settings.js#L8).

### 1.7 (부차적이지만 방치 불가) AI 기능이 언제든 멈출 수 있다
- 모든 서버리스 API가 무인증 공개 상태여서 비용 폭탄이 터지면 단어 분석·문법·채팅 등 AI 기능 전체가 중단되어 학습 루프가 멈춘다. (자세한 내용은 §3)

## 2. 확실한 개선 방향 (우선순위 로드맵)

원칙: **"이미 사용자가 도달하는 화면(복습)의 신뢰를 먼저 복구 → 인출의 질을 높임 → 신규 루프 추가"** 순서. P0 없이는 무엇을 해도 진전이 기록되지 않으므로 상위 기능이 무의미하다.

### P0 — 1주차: 복습 엔진·피드백 루프 수리 (신규 기능 금지, 수리만)
1. **인출 강제**: Review에서 `showAnswer===true`일 때만 평가 버튼을 렌더링. (§1.2)
2. **스케줄러 교체**: 7일 상한·`-10` 영구 제외를 폐기하고, 이미 스키마에 있는 `ease_factor/repetitions/interval_days`로 표준 SM-2(또는 `ts-fsrs`) 전환. 기존 stack 값은 초기 interval로 환산. (§1.1)
3. **죽은 통계 복구**: `handleEvaluation`에서 `updateTodayStats('cards_reviewed')/('cards_correct')` 호출 + `updateReviewResult`에서 `repetitions` 증가. (§1.3)
4. **무결성 일괄 수정**: `getTodayReviewCount`에 `.gt('stack', -10)` 통일, UTC 날짜를 로컬 날짜로 교체, 'Review Again' 실제 구현 또는 제거. (§1.5)
5. **비용 누수 차단**: `api/gemini*`·`whisper`·`screenshot`에 Supabase JWT 검증 + 사용자별 rate limit 추가, `gemini-ws-url` 제거. **키 회전 필수.** (§3)

### P1 — 2~3주차: 카드의 질 (맥락·산출 방향)
6. 단어 저장 시 **원문 문장을 함께 저장**(뷰어에 이미 `findSentenceFromWord` 존재, YouTube는 `segment.text`). 카드 앞면을 **문장 속 빈칸(cloze)**으로 전환.
7. **카드 방향 이원화**: 인식(E→K)과 산출(K→E, 타이핑 채점)을 번갈아 출제. `ai_analysis_json` 없는 highlight는 review_item 생성에서 제외("질문=답" 카드 근절).
8. **세션 구조화**: 세션당 카드 상한(예: 20개) + 종료 시 결과를 `user_stats`에 영구 기록.

### P2 — 4~6주차: 산출 루프·습관 설계
9. **AI 튜터를 "설명 봇"에서 "산출 코치"로**: 복습 종료 직후 "오늘 복습한 단어 3개로 영어 문장을 만들어 보라"는 과제를 채팅에 자동 삽입 + AI 교정 피드백. 기본 응답을 "영어 본문 + 한국어 보조" 혼합으로.
10. **스트릭·일일 목표**: `user_stats`에 날짜별 행이 생기면 연속 학습일 계산 가능. 홈 상단에 스트릭·목표 진행률 노출. Mypage의 접힌 통계를 홈으로 승격.
11. **재참여 트리거**: PWA 푸시 또는 이메일 "오늘 복습 N개" 리마인더(이미 만들어둔 `notification_settings`/`push_tokens` 스키마 활용).

### P3 — 그 이후: i+1
12. 온보딩에 간단한 레벨 자가진단 + 소스 추가 시 텍스트 난이도(어휘 빈도 readability + 단어장 커버리지) 계산 → "적정/어려움" 배지. 복습 정답률이 쌓이면 레벨 자동 갱신.

## 5. UI / UX 문제

**여정 총평**: 핵심 루프(단어 탭→저장→복습)의 앞단은 낙관적 업데이트·스켈레톤 로딩·위치 인식 툴팁 등 완성도가 높다. 그러나 **여정의 뒷단이 무너져 있고, 에러 처리가 전반적으로 침묵(silent catch) 패턴**이라 실패가 무반응 또는 오해를 부르는 빈 상태로 나타난다.

## 6. 대안 접근 (방향 선택지)

복습 엔진 교체(§2 P0-2)에 대한 4가지 선택지:

| 대안 | 개요 | 트레이드오프 |
|------|------|-------------|
| **① FSRS 도입** (`ts-fsrs`) | Anki가 쓰는 검증된 스케줄러. stability/difficulty 컬럼 + Again/Hard/Good/Easy 4단계 | ✅ 검증된 알고리즘, 개인화 ❌ 마이그레이션 근사치, 4단계가 라이트 유저에 마찰, 의존성 추가 |
| **② 최소 수리형 SM-2** | 기존 스키마의 `ease_factor/repetitions/interval_days`로 교과서 SM-2를 40줄 구현, UI 불변 | ✅ 변경 최소, 2버튼 유지, 의존성 0 ❌ 초기 간격 공격적, 개인화 없음, 경계 조건 직접 검증 |
| **③ 세션 중심 재설계** (Duolingo형) | "복습 10 + 새 단어 5 + AI 문장 1 = 오늘의 학습". 홈을 세션 중심으로, 세션 완료가 스트릭 단위 | ✅ 동기 설계·"뭘 할지" 혼란을 한 번에 해결 ❌ 개발 범위 최대(홈 IA 재설계), API 비용 증가, 파워 유저에 강제 구조 마찰 |
| **④ AI 생성 문항 복습** | 단어마다 Gemini가 새 예문 cloze·한→영 작문 생성, 타이핑 답안 AI 채점 | ✅ 인식→산출 전환 + 맥락 다양화, 학습 효과 상한 최고 ❌ 복습 1장당 API 1~2회(비용·지연), **보안 수정(§3) 선행 필수**, AI 오채점 리스크, 오프라인 불가 |
