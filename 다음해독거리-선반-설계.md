# "다음 해독거리" 선반 — 설계서

> 2026-07-04. 설계만, 구현은 아직 안 함.
> 도출 과정: 멀티에이전트 설계 리뷰 — 코드베이스 정밀 매핑(4) → 독립 설계안(3: 최소인프라 MVP / UX우선 / 데이터엔진) → 심판 패널(3: 프로덕트/기술/ROI 렌즈, 라이브 검증 포함). **심판 전원일치로 MVP 안 승리(44·43·44점)**, 아래는 승자 안에 나머지 두 안의 검증된 요소를 접목하고 심판이 적발한 틀린 가정을 교정한 최종안.

---

## 1. 한 줄 정의

홈 화면에 **"다음에 풀어볼 영상"** 가로 선반 하나. 후보는 오직 **사용자가 이미 추가한 유튜브 채널의 새 업로드**(revealed preference)에서 나오고, 탭하면 기존 AddSourceModal의 미리보기→자막확인→(유료 Whisper는 기존 명시 확인) 흐름으로 들어간다. 북극성 = **"다음 해독거리를 찾는 마찰 제거"**. 클릭 극대화 아님.

## 2. 원칙 (설계 전체를 지배)

1. **선반은 제안이지 과제가 아니다** — 보여줄 게 없으면(콜드스타트·전부 추가/숨김·페치 실패) 섹션 자체가 **사라진다**. "새 영상 없음" 류의 빈 상태 문구조차 없다.
2. **선반은 돈을 쓸 수 없다** — 렌더는 무료 메타데이터(RSS+img.youtube.com)만. 추가는 항상 기존 AddSourceModal 동의 흐름 경유. Whisper는 기존 명시적 비용 확인 버튼 뒤에만.
3. **금지 문자열 목록(리뷰에서 기계적으로 잡을 수 있게 개념이 아닌 문자열로)**: "놓친", "밀린", "지금 보세요", 느낌표 붙은 개수("N개의 새 영상!"), 빨간색/배지/카운트 일체. 날짜는 중립 상대시각("3일 전")만.
4. **결정론적 정렬** — 카드가 렌더 사이에 재셔플되어 novelty를 제조하지 않는다. 신선한 목록은 **다음 마운트에서만 적용**(유일한 즉시 적용 경로 = 사용자가 직접 누른 새로고침).
5. **모달을 닫는 것은 거절이 아니다** — 저장 없이 닫아도 카드는 남는다. 숨기기(✕)만이 제거다.

## 4. 데이터 모델 — 테이블 0개, 마이그레이션 0개

1. **`sources.youtube_data` JSONB 확장** (DDL 불필요): `channel_id?: string`(UC+22자), `author_url?: string`, `duration?: number`(lengthSeconds). Phase 2부터 추가 시점에 기록, 레거시 소스는 lazy 백필(아래). TS: database.ts의 `YouTubeData`에 필드 추가.
2. **localStorage** (settings.js snake_case 관례, 읽기는 try/catch):
   - `decode_shelf_cache_v1` = `{ fetchedAt, items: [{videoId, title, channelId, channelName, published, thumbnail, score, parts}] }`
   - `decode_shelf_dismissed_v1` = `{ [videoId]: dismissedAtISO }` — **읽을 때 90일 초과 항목 정리**(UX안 접목; 영구숨김+300개 축출보다 낫다)
   - `decode_shelf_resolved_v1` = `{ [videoId]: channelId | null }` — 채널 해석 메모(실패 null도 캐시, 수동 새로고침 때만 재시도)
   - `shelf_include_shorts` (settings.js 관례, boolean, **기본 false**) — Shorts 노출 여부 사용자 토글(Settings에서 켜고 끔)
3. **캐시 아이템 객체는 나중의 `discovery_candidates` 테이블 컬럼과 호환되는 모양으로 설계**(데이터안 접목) — 푸시 배달이 실제로 일정에 오르면 이 localStorage 캐시를 그 테이블로 옮기는 게 사전 설계된 마이그레이션 경로다(§10).

## 5. 후보 소싱 — 새 무키 프록시 1개: `api/youtube-feed.js`

CORS/OPTIONS 프리앰블은 api/youtube-captions.js:9-14 복사. 시크릿 없음. POST 2모드:

- **`{channelIds: string[]}`** (≤10개): 각 RSS 피드를 `Promise.allSettled`로 서버측 페치, 정규식 파싱(엔트리가 평평해서 XML 라이브러리 불필요). **엔트리의 `<link rel="alternate">` href가 `/shorts/`면 `isShort: true` 마킹**(서버측, Phase 1부터 — 확실한 판별법). 채널별 실패는 null로 반환(502 아님). 반환: `{ channels: { [channelId]: { channelName, items: [...] } } }`.
- **`{resolveVideoId}`**: watch 페이지를 tracksFromWatchPage와 동일 레시피(UA+CONSENT 쿠키)로 페치, `"channelId":"(UC[\w-]{22})"` + `"lengthSeconds":"(\d+)"` 추출. 레거시 소스 백필용.

**예산**: 리프레시 1회 = Vercel 호출 1 + 서버측 피드 페치 N(현실적으로 <15, 각 30-50KB) — **기본 ~10초 함수 예산 안**(60초 가정 금지). 리프레시는 TTL로 ~2회/일. 해석 백필은 레거시 영상 수만큼의 1회성 버스트 후 영원히 0.

**채널 발견(클라이언트)**: `getYouTubeSourceMeta()`를 source.ts에 추가 — `select('id, youtube_data, pinned').eq('type','youtube')` (기존 getSources는 youtube_data를 일부러 제외하므로 별도 경량 쿼리). channel_id 없는 레거시는 resolve 1회→ `decode_shelf_resolved_v1` 메모 + `updateSource`로 youtube_data에 **write-back**(read-modify-write, 기존 객체 spread 필수 — 통째 교체 금지). 실패(null)도 캐시하되 수동 새로고침 경로에서 재시도 가능하게.

**범위**: 전사-키워드 검색은 Phase 3(아래, Gemini 캐시 경유 무키 경로 확보). 웹 RSS는 보류, PDF는 제외(합의대로).

## 6. 랭킹 v1 — signal registry (데이터안의 구조 접목)

`rankShelf(candidates, sourcesMeta, dismissed)` 순수 함수, src/services/shelf.js.

**하드 필터**: 이미 추가된 videoId 제거 → dismissed 제거 → **`shelf_include_shorts`=false(기본)이면 `isShort` 제거**(사용자가 Settings에서 켜면 Shorts도 노출 — "둘 다 가능"). `isShort` 판별은 §5의 RSS link href로 서버측에서 이미 마킹됨.

**점수 = Σ wᵢ·sᵢ**, 각 신호는 [0,1], **신호가 null이면 가중치 재정규화**(나중 difficultyFit이 순수 추가가 되는 열쇠):

| 신호 | v1 가중치 | 정의 |
|---|---|---|
| `channelAffinity` | 2.0 | log2(1+해당 채널 소스 수) + (핀 소스 있으면 +1), max 정규화 |
| `freshness` | 1.0 | clamp(1 − ageDays/30, 0, 1) — 의도적으로 완만·단조(긴급함 프레이밍 불가) |
| `keywordMatch` | (Phase 3) | 후보 제목 토큰 ∩ youtube_data.keywords 합집합 |
| `difficultyFit` | (later) | §11 — 지금은 parts 객체의 예약 슬롯 |

각 후보에 `parts: {channelAffinity, freshness, keywordMatch, difficultyFit}` 기록(score_breakdown 모양 — 나중 테이블 컬럼과 호환).

**다양성 가드**: 채널당 최대 2장, 총 **3장 상한**(v1 결정; Phase 4 난이도 랭킹이 붙어 랭킹을 신뢰할 수 있게 되면 **1장으로 수렴** — "카드 수 = 랭킹 신뢰도"), "더보기" 없음. **정렬 결정론**: score desc → published desc → videoId. **콜드스타트**: 채널 1개면 그 채널 최신 미추가 업로드가 그대로 정답(저하 모드 아님); 유튜브 소스 0개면 섹션 부재(추가 유도 CTA 금지).

## 7. UI 스펙

**배치**: Home.jsx의 `.continue-section` 닫힘(현 213행)과 `.source-section`(215행) 사이, non-error fragment 안. 가드: `!loading && statusFilter === 'all' && !searchQuery.trim() && shelfItems.length > 0` (continue-section 관례 + 자기은폐).

**구조**: `.section-header`(h2 = `home.shelfTitle`, 카운트 없음) + 우측 조용한 텍스트 버튼 `.shelf-refresh`. 아래 `.shelf-row` — `.filter-tabs`의 overflow-x:auto 관용구 그대로. **스타일은 index.css에 append**(home.css 신설 금지).

**카드 `.shelf-card`** (`.source-card` 재사용 금지 — 3/4 비율 상속): flex:0 0 auto, width ~172px. 16:9 `.shelf-thumb`(lazy img, onError→▶ 글리프 다크 플레이스홀더) / `.shelf-title` 2줄 클램프 / `.shelf-meta` = `"{channelName} · 3일 전"` (--color-text-sub) / 우상단 24px `.shelf-dismiss` ✕(상시 60% 불투명, stopPropagation).

**카피 (ui.json home 네임스페이스에 {en,ko} 실등록)**:
- `shelfTitle`: en "Next to Decode" / ko **"다음에 풀어볼 영상"** (UX안 카피 채택)
- `shelfRefresh`: en "Refresh" / ko "새로 보기"
- `shelfDismiss`(aria): en "Skip this video" / ko "이 영상 건너뛰기"
- 상대시각: "오늘"/"어제"/"{n}일 전"/"{n}주 전" — 사실 서술만

**인터랙션**:
- 카드 탭 → `AddSourceModal`에 `initialUrl` prop으로 오픈. **⚠️ 구현 함정(반드시 이렇게)**: 모달엔 useEffect가 없고 44행 early-return → `key={initialUrl}` 리마운트 또는 mount-keyed effect로 `handleLinkChange(initialUrl)`를 **정확히 1회** 발화, linkSeqRef 가드 보존. 사용자는 익숙한 미리보기(썸네일+제목+무료 자막 확인) → 기존 저장/Whisper-비용확인 선택지에 착지.
- ✕ 탭 → dismissed 기록 + 즉시 제거. 확인 없음. (Phase 2에 일시적 "숨겼어요 · 되돌리기" 칩 옵션 — UX안 접목)
- 추가 성공 → 기존 onSuccess/loadData 후 캐시 재랭크(네트워크 없이)로 해당 카드 자연 탈락.
- 저장 없이 모달 닫기 → 카드 유지(거절 아님).

**상태**: 캐시 없음+로딩 → 스켈레톤 3장(`.skeleton-shimmer`) / 캐시 있음 → 즉시 렌더, 스피너 없음(SWR) / 페치 실패 → 스테일 캐시 있으면 조용히 표시, 없으면 **섹션 부재**(console.warn만 — `.load-error-state`는 사용자 소유 데이터 전용, 선반은 빌린 자리) / 필터·검색 중 → 숨김.

## 8. 새로고침 전략

localStorage SWR. 마운트 시 `fetchedAt` < **TTL 12h**면 캐시 렌더 + **네트워크 0**(랭킹 입력도 캐시에 동봉 — Supabase 호출조차 없음). 스테일이면: 캐시 즉시 렌더 → 백그라운드로 meta 쿼리→해석→피드→랭크→캐시 기록, **적용은 다음 마운트**(주문 안정성). 수동 "새로 보기"만 즉시 적용 — 단 **10분 창 내 재탭은 조용한 no-op**(데이터안 접목: 새로고침 버튼이 슬롯머신 레버가 되는 것 방지) + single-flight 가드. 타이머·포커스 리프레시·폴링 없음.

## 9. 가드레일 요약 (구현 시 체크리스트)

카운트/배지/빨강 금지 · **3장 상한**, 채널당 2장, 무한스크롤 금지 · 금지 문자열 목록(§2-3) · freshness 가중치 상한(랭킹이 recency ticker화 불가) · 숨김은 1탭·무질문·**90일간 유지 후 재부상**(그 기간 피드에 재등장해도 dismissed 유지 — **병합의 불변식**) · 빈/에러/콜드스타트 = 섹션 부재 · 추가는 항상 기존 동의 흐름 · 후보는 이미 추가한 채널에서만(트렌딩/관련영상/탐색 주입 금지) · 결정론 정렬 · 푸시 없음(범위 밖).

## 10. 구현 단계

### Phase 1 — 출시 가능한 선반 (한나절, 4-6h + 모달 배선 여유)

### Phase 3 — 전사-키워드 후보 (합의 범위 복귀)
**3a (무키, 먼저)**: Gemini 키워드 추출 — 소스당 **딱 1회**(가드: `youtube_data.keywords` 부재 시만), 프롬프트 = 제목 + 전사 앞 ~8k자, 영어 검색구 5개 JSON, 기존 /api/gemini 프록시로 → youtube_data.keywords 캐시 → `keywordMatch` 신호 가동(같은 채널이라도 관심 주제 영상이 위로).
**3b (선택, 키 1개)**: api/youtube-search.js — YouTube Data API search.list 프록시(신규 `YOUTUBE_API_KEY`; 검색당 100유닛, ≤5키워드×2리프레시/일 = 1,000/10,000 무료쿼터, 10배 여유). 결과는 같은 파이프라인 합류, reason 칩 "'{keyword}' 관련 영상". **RSS만으로 부족하다고 실사용에서 확인된 뒤에만.**

### Phase 4 (later — 지금은 설계만, 데이터안의 스펙을 참조 설계로 채택)
`difficultyFit` 신호 가동: v1 상위 ~10 후보만 기존 /api/youtube-captions로 **무료** 자막 페치(후보에 Whisper 절대 금지) → (a) unknownRate = 전사 단어 타입 중 미지 비율, known-set = 저장 어휘 − **ease_factor<1.8은 아직-미지로 계산** + 상위 2k 빈도 리스트를 기지 바닥으로 (b) 후보 WPM (c) 사용자 컴포트 밴드 = 해독한 소스들의 중앙값(Whisper 소스는 `segments[].words`로 단어수준 조음률) → **target unknownRate ≈ 5-8%(i+1/flow 밴드) 가우시안** × WPM 거리 클램프. 자막 없는 후보는 null → 가중치 재정규화 폴백. parts 객체가 이미 있으므로 **스키마·파이프라인 변경 0**.

## 11. 푸시 배달로 가는 길 (지금 짓지 않지만 막히지 않게)

푸시가 실제로 일정에 오르는 시점에: 데이터안의 `discovery_candidates` 테이블(UNIQUE(user_id,video_id), status enum shown/dismissed/added — **upsert가 status를 절대 덮어쓰지 않는 규칙**, score_breakdown JSONB)을 그대로 마이그레이션 타깃으로. §4의 캐시 아이템이 컬럼 호환이므로 이관은 기계적. 서버 cron이 같은 랭킹을 읽는다. **그 전에는 만들지 않는다**(1인 앱에서 localStorage로 충분한 걸 테이블로 짓는 건 증거에 앞선 인프라).

## 12. 리스크

1. RSS는 비공식-인접(2005년부터 안정, 오늘 검증됨)이나 구글이 예고 없이 막을 수 있음 — 선반은 우아하게 숨지만 기능은 피드와 운명공동체.
2. channelId 해석은 자막 스크레이퍼와 같은 datacenter-IP 취약성 공유(_debug 필드가 존재하는 이유) — 실패는 null 캐시, 수동 새로고침에서 재시도.
3. 레거시 `channel:'Unknown'` 소스(noembed 장애 때 저장)는 조용히 후보 기여 0 — 수용 가능하나 비가시적.
4. localStorage 전용 상태 = 기기 간 비동기화(1인·주로 단일 기기라 수용; §11이 탈출로).
5. youtube_data write-back은 동시 쓰기에 취약 — 병합은 항상 기존 객체 spread.
6. api/youtube-feed.js도 기존 7개처럼 무인증 ACAO:* 라우트 — 시크릿 없고 비용 0이라 기존 posture와 동일(그 posture 자체는 별도 이슈로 이미 플래그됨).
7. 미래의 affinity 고도화(annotations/chat_logs 집계) 시 **PostgREST 기본 ~1000행 상한** 주의 — source_id 컬럼만 select.

## 13. 결정 현황

- **✅ 1. 카드 탭 → 모달 경유 (확정)** — 기존 AddSourceModal 동의 흐름 그대로. 원탭 추가는 라이브러리 호딩·비의도적 커밋을 유발하므로 기각.
- **✅ 2. 카드 3장 (확정)** — v1은 3장(멍청한 v1 랭킹을 헷지). Phase 4 난이도 랭킹 후 1장으로 수렴. 6장은 채널 2-3개 초기엔 못 채우고 피드 느낌만.
- **✅ 3. 숨김(✕) 지속 = 90일 후 재부상 (확정)** — 취향은 안정적이나 바빠서 넘긴 건 재고 여지; `decode_shelf_dismissed_v1`을 읽을 때 90일 초과분 정리(§4).
- **✅ 4. Shorts = 사용자 설정, 둘 다 가능 (확정)** — **기본 제외**, Settings에 "Shorts 포함" 토글(`shelf_include_shorts`, 기본 false) 추가. 서버는 RSS link href로 `isShort`를 항상 마킹, 필터는 설정을 따름(§6). 세로 썸네일이 16:9 카드에서 깨져 보이는 문제는 포함 시 감수 or 카드가 crop.
- **✅ 5. to_read 성공 마커 = 마커만 심기, UI 없음 (확정)** — 추천 줄에서 추가한 소스에 조용히 `to_read=true`(1필드). "선반이 실제로 해독을 먹였나"를 나중에 궁금하면 DB로 확인. 대시보드/UI는 안 지음 = 안 볼 측정은 안 만듦. (성공 지표 = 클릭이 아니라 **추천발 소스의 해독 여부**.) 구현: 카드 탭 시 모달에 `fromShelf` 마커 전달 → createYouTubeSource가 to_read=true로 insert(현재 to_read는 미기록 dead 컬럼이라 write 경로 신설).
